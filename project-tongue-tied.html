01 Overview

Tongue Tied is a two-player cooperative mini-game built in Python for a SUTD class project (SC08, 2022). Two players share one keyboard, control two frogs, and must navigate a river together — all while staying physically close to each other on screen. The game was designed by a team of six students as an interactive exploration of communication and coordination.

Context
SUTD SC08 class project, December 2022. Team of 6. Theme: use software to solve an interesting problem.
Tech Stack
Python · turtle graphics · OOP · AABB collision detection · game state machine
02 Design Goal

How do you force two people to communicate — not through instructions, but through the rules of the game itself?

The core design question was social: many games are called "co-op" but can be played without saying a word. We wanted a mechanic where silence literally causes failure. The answer was a distance constraint — if the two frogs drift more than 60 pixels apart in either axis, movement is blocked. This makes communication the gameplay, not just a nice-to-have.

03 Gameplay

Both players sit at the same keyboard. Player 1 uses arrow keys; Player 2 uses WASD. Both frogs must reach the lilypad together, avoid moving logs, and survive 4 levels with shared lives. Press K to start or restart, P to quit.

04 What Makes It Interesting
The mechanic is the message. The 60px leash isn't just a technical rule — it encodes a social idea. You can't go faster than your partner, and your partner can't go faster than you. The rope forces empathy.
One keyboard, two players. Sharing input forces physical proximity, which changes the social dynamic of play. This is deliberate — the game is designed for pairs sitting together, not online.
Shared lives create shared stakes. Losing a life hurts both players equally. There's no pointing fingers — success and failure are genuinely collective.
The name is the concept. "Tongue Tied" — two tongues, one knot. The frog theme is playful, but the name captures what the game is really about: what happens when two people have to untie themselves together.

06 Technical Notes
Architecture
OOP class hierarchy: Sprite → Player / Car / Goal. Each entity manages its own state, rendering, and update logic. A shared pen turtle handles all canvas drawing.
Collision
AABB (Axis-Aligned Bounding Box) — checks whether the absolute difference between two objects' x and y coordinates falls within their combined half-widths and half-heights.
Distance Constraint
Each Player.up/down/left/right method calls far_apart_x() or far_apart_y() before moving, returning early if the signed separation exceeds ±60px. This asymmetric check means both players can lock each other.
Game Loop
Single while True loop with gamestate string ("Start" / "Game" / "Dead" / "Win" / "Quit"). wn.tracer(False) + wn.update() gives manual frame control for smooth rendering.
Difficulty Scaling
Each Car has a base dx and an accumulated speed float. On each level-up, speed_up() increments |speed| by 0.1, making logs progressively faster without changing the base movement delta.
